Sincronismo e Multifiliais
Arquitetura
Railway (nuvem) = servidor central passivo. Cada farmacia roda backend + PostgreSQL local. Sync bidirecional: PC local -> Railway -> outros PCs.
VersaoSync
Cada registro tem um campo VersaoSync (timestamp em milissegundos, monotonicamente crescente). Usado para detectar alteracoes pendentes de envio/recebimento.
FilialOrigemId
Identifica de qual farmacia/servidor o registro foi criado. Fonte primaria: Sync:FilialLocalId da config do servidor. PUSH envia apenas registros onde FilialOrigemId == FilialLocalId. PULL filtra registros de outras filiais (evita echo).
SuspenderAutoSync
Flag no AppDbContext. Quando true, SaveChangesAsync NAO incrementa VersaoSync. Usado APENAS no PULL local para preservar a versao do central e evitar loop infinito. O Railway NAO usa essa flag.
Ciclo de Sync (a cada 30s)
1. Autentica no Railway como usuario SISTEMA (senha rotativa diaria). 2. Para cada tabela: PUSH (envia locais) depois PULL (recebe remotos). 3. Apos PULL, reseta sequences do PostgreSQL (evita conflito de PK). 4. Atualiza SyncControle com versoes enviadas/recebidas.
Resolucao de Conflitos
Last-write-wins baseado em AtualizadoEm. Se PC1 e PC2 alteram o mesmo registro, o ultimo a sincronizar ganha. Futuramente: tabela SyncConflitos para registrar e revisar.
Tabelas: Global vs Por Filial
Global: replica inteira, todos veem igual (Fabricantes, Substancias, Produtos). Por Filial: cada filial tem seus registros filtrados por FilialId (Estoque, Caixa, Vendas, Configuracoes). Definido no Dicionario de Dados.
Tabelas que NAO replicam
Configuracoes (cada filial define a sua), LogsErro (local), SyncControle (controle interno), DicionarioTabelas/Revisoes (ferramenta dev).
Campos unicos no sync
CPF, CNPJ e Login devem ser validados antes de aplicar registros remotos. Se PC1 cadastra CPF "123" e PC2 tambem, gera conflito. Definido no Dicionario de Dados (coluna Unico).
Reset de Sequences
Apos PULL, o sync reseta a sequence auto-increment de cada tabela para MAX(Id)+1. Necessario porque registros sincronizados vem com IDs explicitos que o PostgreSQL nao rastreia.
Configs por PC
Arquivos appsettings.pc1.json e pc2.json (no .gitignore, nao sobem pro Git). Contem: Sync:Habilitado, IntervaloSegundos, FilialLocalId, UrlCentral e Cassi:ApiKey.
Queda de internet
O sistema continua funcionando normalmente offline (banco local). O sync falha silenciosamente e tenta novamente no proximo ciclo. Quando a internet volta, sincroniza automaticamente de onde parou.
Cassi (Assistente IA)
Como funciona
Backend proxy para OpenAI (gpt-4o-mini). API key no appsettings.pcX.json (nunca no Git). System prompt fixo + instrucoes dinamicas do Dicionario de Dados.
Instrucoes IA
Definidas no Dicionario de Dados por tabela e por campo. A Cassi le antes de cada conversa. Ao atualizar o DD, a Cassi fica mais inteligente automaticamente.
Acoes automaticas
A Cassi pode abrir telas automaticamente. Responde em JSON com {{ '{' }} mensagem, acao {{ '}' }}. Se acao tem uma rota, o frontend navega automaticamente.
Dicionario de Dados
Proposito
Documentacao viva da estrutura do banco. NAO altera o banco, apenas registra como deveria ser. Serve como base para implementar sync, validacoes e multifiliais.
Por tabela
Escopo (Global/Por Filial), Replica (Sim/Nao), Instrucao IA. Define o comportamento da tabela no sincronismo.
Por campo
Revisado, Obrigatorio, Unico, Instrucao IA, Observacao. Quando todos os campos estao revisados, a tabela fica com borda verde.
Backup JSON
Cada alteracao grava automaticamente em ContextDocuments/dicionario-dados.json. Vai pro Git como backup. Se perder o banco, importa pelo endpoint POST /api/dicionario-dados/importar.

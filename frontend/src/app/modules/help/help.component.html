Sincronismo e Multifiliais
@if (isAberta('sync-doc')) {
Sync v2 - Fila de Operacoes
Cada INSERT/UPDATE/DELETE gera um registro na tabela SyncFila automaticamente. O sync envia essas operacoes para o Railway, que distribui para as outras filiais. DELETE replica naturalmente.
Identificadores: Id vs Codigo
Id = tecnico, PK local, auto-increment. Pode ser diferente entre PCs. Codigo = visivel, formato "FilialCodigo.Sequencial" (ex: "2.115950"). Unico global. O sync busca registros por Codigo, nao por Id.
FilialOrigemId
Identifica qual farmacia/PC criou o registro. Automatico, invisivel para o usuario. Usado pelo sync para filtrar operacoes (nao puxar as proprias).
Ciclo de Sync
A cada 30s: 1. Autentica no Railway como SISTEMA. 2. PUSH: envia operacoes pendentes da SyncFila. 3. PULL: busca operacoes de outras filiais e aplica localmente (INSERT por Codigo, UPDATE por Codigo, DELETE por Codigo).
Tabelas: Global vs Por Filial
Global: qualquer filial cria/edita (Fabricantes, Substancias, Produtos). Por Filial: cada filial tem seus registros (Estoque, Vendas, Caixa). NAO replicam: Configuracoes, SyncFila, SequenciaLocal, DicionarioTabelas.
Queda de internet
Sistema funciona 100% offline. Operacoes ficam na SyncFila como pendentes. Quando internet volta, envia tudo automaticamente.
Tela Sincronismo
Mostra fila com filtros (periodo, status, tabela), paginacao, cards de status. Permite forcar envio e limpar antigos.
}
Cassi (Assistente IA)
@if (isAberta('cassi')) {
Como funciona
Backend proxy para OpenAI (gpt-4o-mini). API key no appsettings.pcX.json (nunca no Git). System prompt fixo + instrucoes dinamicas do Dicionario de Dados.
Instrucoes IA
Definidas no Dicionario de Dados por tabela e por campo. A Cassi le antes de cada conversa. Ao atualizar o DD, a Cassi fica mais inteligente automaticamente.
Acoes automaticas
A Cassi pode abrir telas automaticamente. Responde em JSON com {{ '{' }} mensagem, acao {{ '}' }}. Se acao tem uma rota, o frontend navega automaticamente.
}
Dicionario de Dados
@if (isAberta('dd')) {
Proposito
Documentacao viva da estrutura do banco. NAO altera o banco, apenas registra como deveria ser. Serve como base para implementar sync, validacoes e multifiliais.
Por tabela
Escopo (Global/Por Filial), Replica (Sim/Nao), Instrucao IA. Define o comportamento da tabela no sincronismo.
Por campo
Revisado, Obrigatorio, Unico, Instrucao IA, Observacao. Quando todos os campos estao revisados, a tabela fica com borda verde.
Backup JSON
Cada alteracao grava automaticamente em ContextDocuments/dicionario-dados.json. Vai pro Git como backup. Se perder o banco, importa pelo endpoint POST /api/dicionario-dados/importar.
}
Pendencias para Producao
@if (isAberta('pendencias')) {
SYNC_PROFILE automatico
Em producao, o SYNC_PROFILE sera configurado pelo instalador/setup da farmacia. Nao sera necessario setar manualmente via PowerShell.
Esconder bloco Dev
O bloco Dev (Help, Dicionario de Dados) deve ser escondido em producao. Visivel apenas em ambiente de desenvolvimento.
Certificado SSL
Configurar certificado SSL valido para o backend local (atualmente usa DangerousAcceptAnyServerCertificateValidator).
Senha SISTEMA
Revisar a chave SistemaKey usada para gerar a senha rotativa do usuario SISTEMA. Em producao deve ser unica por cliente.
API Key Cassi
Definir estrategia de distribuicao da API key do ChatGPT para cada instalacao. Considerar limites de uso e custo.
Backup automatico
Implementar rotina de backup automatico do PostgreSQL local de cada farmacia.
}
Checklist: Criar Nova Tabela
@if (isAberta('checklist')) {
1. Entidade (Domain)
Criar classe em backend/ZulexPharma.Domain/Entities/. Herdar de BaseEntity (Id, Codigo, Ativo, CriadoEm, AtualizadoEm, FilialOrigemId). Definir propriedades e navigation properties.
2. DbContext (Infrastructure)
Adicionar DbSet no AppDbContext.cs. Configurar no OnModelCreating: HasKey, UseIdentityByDefaultColumn, HasMaxLength, IsRequired, HasIndex (Unique), relacionamentos (HasOne/WithMany + OnDelete).
3. Migration
dotnet ef migrations add NomeDaMigration --project ZulexPharma.Infrastructure --startup-project ZulexPharma.API IMPORTANTE: parar o backend antes de rodar. A migration roda automaticamente no proximo startup (MigrateAsync).
4. Dicionario de Dados
Adicionar a tabela no DatabaseSeeder.cs (seed de DicionarioTabelas) com Escopo, Replica e InstrucaoIA. Isso faz a tabela aparecer no DD automaticamente com as instrucoes corretas para a Cassi. context.DicionarioTabelas.Add(new DicionarioTabela {{ '{' }} Tabela = "NomeTabela", Escopo = "global", Replica = true, InstrucaoIA = "Descricao para a Cassi" {{ '}' }});
5. Sync
Adicionar no mapa _tiposPorTabela do SyncBackgroundService.cs para que a tabela participe do sincronismo. Se a tabela NAO deve replicar, adicionar na lista _tabelasSemSync do AppDbContext.cs.
6. Service + Interface
Criar IService em Application/Interfaces/ e Service em Infrastructure/Services/. Registrar no Program.cs com AddScoped. Implementar: ListarAsync, ObterAsync, CriarAsync, AtualizarAsync, ExcluirAsync.
7. Controller (API)
Criar em API/Controllers/. Atributos: [Authorize], [ApiController], [Route("api/[controller]")]. Endpoints: GET (listar), GET/id (detalhe), POST (criar), PUT/id (atualizar), DELETE/id (excluir). Padrao de resposta: {{ '{' }} success, data, message {{ '}' }}.
8. DTOs
Criar em Application/DTOs/NomePasta/. Normalmente: ListDto (para grid), DetalheDto (para editar), FormDto (para criar/atualizar).
9. Frontend
Criar componente em frontend/src/app/modules/nome/. Arquivos: component.ts, component.html, component.scss. Registrar rota em erp-shell.routes.ts. Adicionar tile na Home (dashboard.component.ts) ou em Outros Cadastros.
Estrutura de IDs
IDs sao globais por faixa de filial (1 bilhao por filial). Filial 1: IDs 1.000.000.001+, Filial 2: 2.000.000.001+. Configurado automaticamente no startup. FKs funcionam direto entre filiais sem remapeamento.
}

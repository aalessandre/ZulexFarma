Sincronismo e Multifiliais (Arquitetura Completa)
@if (isAberta('sync-doc')) {
Visao Geral
Cada loja roda em servidor local (localhost + PostgreSQL local). O Railway e o banco central na nuvem com dados consolidados de todas as filiais. Fluxo: Filial faz CRUD local → PUSH para Railway → Railway aplica no proprio banco E enfileira na SyncFila → Outras filiais fazem PULL.
Fluxo do Sync (ciclo a cada 30s)
1. Autentica no Railway como usuario SISTEMA (senha rotativa diaria via SHA256). 2. PUSH: envia operacoes pendentes da SyncFila local para POST /api/sync/enviar. 3. Railway recebe, ORDENA por dependencia, APLICA no proprio banco (AplicandoSync=true), e enfileira para outras filiais. 4. PULL: filial busca GET /api/sync/receber?filialId=X&ultimoId=Y, ordena por dependencia, e aplica localmente.
Railway como Banco Consolidado
O Railway NAO e apenas um relay de mensagens. Ele APLICA cada operacao recebida no proprio banco PostgreSQL. Isso garante que o Railway tem TODOS os dados de TODAS as filiais. O banco do Railway serve para: consultas cross-filial, dashboards, apps externos. A flag AplicandoSync=true impede que a aplicacao gere novos registros na SyncFila (evita loop infinito).
Ordem de Processamento (Blindagem de Dependencias)
Operacoes sao ordenadas por nivel de dependencia ANTES de aplicar, evitando erros de FK. Para INSERT/UPDATE: pais primeiro (nivel menor). Para DELETE: filhos primeiro (nivel maior). A ordenacao e feita tanto no Railway (ao receber PUSH) quanto nas filiais (ao fazer PULL).
Niveis de Dependencia (SyncApplicator.GetOrdemTabela)
Nivel 0 (sem dependencia): Filiais, UsuariosGrupos, Ncms. Nivel 1 (depende de 0): Pessoas, Fabricantes, Substancias, GruposPrincipais, GruposProdutos, SubGrupos, Secoes, ProdutoFamilias, ProdutosLocais, NcmFederais, NcmIcmsUfs, NcmStUfs, UsuariosGruposPermissao. Nivel 2 (depende de 1): Colaboradores, Fornecedores, PessoasContato, PessoasEndereco, Produtos. Nivel 3 (depende de 2): Usuarios, ProdutosBarras, ProdutosMs, ProdutosSubstancias, ProdutosFornecedores, ProdutosFiscal, ProdutosDados. Nivel 4 (depende de 3): UsuarioFilialGrupos. Nivel 5 (locais): LogsAcao, LogsErro. Fallback desconhecido: 10.
Identificadores: IDs Globais por Faixa de Filial
Cada filial tem faixa exclusiva de 1 bilhao de IDs. Filial 1: 1.000.000.001+, Filial 2: 2.000.000.001+, Filial 3: 3.000.000.001+. Configurado automaticamente no startup via ALTER SEQUENCE RESTART WITH. FKs funcionam direto entre filiais (sem remapeamento). Colisao: praticamente impossivel.
SyncGuid (coluna auxiliar)
Toda entidade BaseEntity tem um campo SyncGuid (UUID, auto-gerado). NAO e PK, NAO tem FK. Serve para reconciliacao futura (ex: restaurar backup, resolver conflitos). Default no banco: gen_random_uuid(). Index criado em todas as tabelas.
Codigo Visivel
Formato "FilialCodigo.Sequencial" (ex: "1.523", "2.1089"). Gerado automaticamente por tabela via SequenciasLocais. Unico global. NAO gerado quando AplicandoSync=true (registros recebidos por sync ja vem com Codigo).
Regra Fundamental: Todas as filiais veem tudo
TODAS as filiais veem dados de TODAS as filiais. TUDO replica, incluindo Configuracoes, LogsAcao e LogsErro. Unicas excecoes: SyncFila e SequenciasLocais (infraestrutura de sync). DicionarioTabelas/Revisoes/Relacionamentos nao herdam BaseEntity, sao identicos pelo seed.
Tabelas Globais vs Por Filial
Globais (replicam para todas): Produto, ProdutoBarras, ProdutoMs, ProdutoSubstancia, Fabricante, GrupoPrincipal, GrupoProduto, SubGrupo, Secao, NCM, Substancia, Fornecedor, Pessoa, Colaborador, GrupoUsuario, Filial, Configuracoes, LogsAcao, LogsErro. Por Filial (replicam, dados variam por filial): ProdutoDados, ProdutoFiscal, ProdutoFornecedor, ProdutosLocais. NAO replicam (infraestrutura de sync): SyncFila, SequenciasLocais. Nao participam do sync (nao herdam BaseEntity): DicionarioTabelas, DicionarioRevisoes, DicionarioRelacionamentos.
ProdutoDados Multi-Filial
Ao criar um produto, o sistema cria automaticamente um registro ProdutoDados (zerado) para CADA filial ativa. Isso garante que toda filial tenha precos/estoque para editar. O SyncDados nunca deleta ProdutoDados de outra filial ao editar — apenas atualiza os que vieram no DTO.
Regra de Propagacao de Preco
Configuracao: produto.preco.regra (editavel em Configuracoes). Valores: "perguntar" = modal pergunta ao usuario quais filiais, "todas" = aplica automaticamente para todas, "atual" = apenas filial logada. Campos propagados: ValorVenda, ValorPromocao, Markup, ProjecaoLucro, DescontoMinimo, DescontoMaxSemSenha, DescontoMaxComSenha, Comissao, ValorIncentivo.
Tratamento de Erros no Sync
FK violation (23503): mensagem legivel indicando registro pai ausente. PK/Unique violation (23505): registro duplicado. Erros sao registrados na SyncFila local com campo Erro preenchido (visiveis no painel). O ponteiro AVANCA mesmo em falha para nao ficar em loop. Entries com erro sao detached do ChangeTracker.
Queda de Internet
Sistema funciona 100% offline. Operacoes ficam na SyncFila como pendentes (Enviado=false). Quando internet volta, envia tudo automaticamente no proximo ciclo. O Railway tambem acumula operacoes para filiais offline — ao reconectar, a filial puxa tudo pendente.
Arquivos-Chave do Sync
SyncApplicator.cs: logica compartilhada (AplicarOperacaoAsync, GetOrdemTabela, ResolverTipo, LimparNavigations, BuscarPorId). Usado pelo Railway e pelas filiais. SyncBackgroundService.cs: background service das filiais (Enviar/Receber a cada 30s). SyncController.cs: endpoints do Railway (POST enviar, GET receber, GET status, POST resetar, POST limpar). AppDbContext.SaveChangesAsync: intercepta CRUDs e gera SyncFila automaticamente (exceto quando AplicandoSync=true).
Checklist: Adicionar Tabela ao Sync
1. Adicionar no mapa _tiposPorTabela do SyncApplicator.cs (["NomeTabela"] = typeof(Entidade)). 2. Adicionar no GetOrdemTabela com o nivel correto de dependencia. 3. Se a tabela NAO deve replicar, adicionar na lista _tabelasSemSync do AppDbContext.cs. 4. Registrar no DatabaseSeeder o DicionarioTabela com Escopo e Replica corretos.
Tela Sincronismo
Mostra fila com filtros (periodo, status, tabela), paginacao, cards de status (pendentes, enviados, recebidos, erros). Permite forcar envio, resetar ponteiro de recebimento (rebuscar tudo), e limpar registros antigos.
}

Lançamento de Compras
@if (isAberta('compras')) {
Visao Geral
Modulo de lancamento de notas fiscais de entrada via importacao de XML NF-e. Acesso pelo menu: Compras > Lancar Compras (/erp/lancar-compras).
Etapas do fluxo
O lancamento de compras funciona em etapas controladas pelo campo Status da tabela Compras:

1. Pre-Entrada — Importacao do XML e vinculacao de produtos
2. Precificacao — Ajuste de precos com comparativo (CC/CM anterior vs atual)
3. Conferencia — Bipe de codigo de barras, controle de fracao, sinalizadores
4. Finalizacao — Duplicatas + Lotes/Vencimentos + atualiza estoque/custos/precos
Cancelada — Nota cancelada (reverte estoque se ja finalizada)
Importacao de XML (3 fontes)
1. Importar Pasta: Botao "Importar XML", seleciona arquivo .xml local da NF-e
2. Buscar SEFAZ: Consulta DistribuicaoDFe por NSU, cache em SefazNotas, manifestacoes
3. Chave NF-e: Consulta consChNFe por chave 44 digitos

— Backend parseia XML (namespace http://www.portalfiscal.inf.br/nfe)
— Verifica duplicata pela ChaveNfe (44 digitos, unique)
— Cria fornecedor automaticamente se CNPJ nao existe (com endereco e telefone do XML)
— Cria registros em Compras + ComprasProdutos + ComprasFiscal
— Parser suporta campos IBS/CBS da reforma tributaria
— XmlConteudo salvo no banco com [JsonIgnore] (nao replica entre filiais)
Etapa 1: Pre-Entrada
Auto-vinculacao:
— Busca por codigo de barras (ProdutosBarras + campo principal)
— Busca por CodigoProdutoFornecedor (ProdutosFornecedores)
— Botao "Atualizar" re-executa auto-vinculacao (POST /api/compras/id/re-vincular)

Vinculacao manual:
— Duplo click na linha abre modal de vinculacao
— Busca silenciosa por barras ao abrir (vincula automatico se encontrar)
— Busca por nome com debounce 400ms (minimo 3 caracteres)
— Ao vincular: insere barras em ProdutosBarras + cria ProdutosFornecedores
— Modal mostra botao Desvincular se item ja vinculado

Cadastro de produto novo:
— Botao "Cadastrar" na modal salva dados do XML no sessionStorage
— Abre Gerenciar Produtos em modo Adicionar com dados pre-preenchidos:
  Nome, Barras, NCM (lookup), CEST, Fornecedor, Fiscal (origem, CST, aliquotas),
  Precos (unitario, ST, custo medio), Registro MS (codigo ANVISA)
— Estado da compra preservado via sessionStorage ao navegar entre abas

Status na lista:
— Pre-Entrada (amarelo) = itens pendentes de vinculacao
— Pre-Entrada OK (verde) = todos os itens vinculados
— Checkbox habilitado apenas para notas com Pre-Entrada OK
Etapa 2: Precificacao
Grid comparativo:
— CC Anterior (ultima compra unitario) vs CC Atual (ValorItemNota/Quantidade)
— CM Anterior (custo medio atual) vs CM Atual (simulacao media ponderada)
— Variacoes % com cores (verde = reducao, vermelho = aumento)

Campos editaveis interligados:
— Sug.Vlr.Venda, %Proj.Lucro, %Markup — alterar um recalcula os outros
— Combobox Base de Calculo: Custo Compra ou Custo Medio
— Formulas: Markup = ((venda-custo)/custo)x100, Projecao = ((venda-custo)/venda)x100
— Projecao limitada a 100% com aviso ao usuario

Botoes de acao:
— Acatar PMC Nota (PrecoMaximoConsumidor do XML)
— Acatar PMC AbcFarma (busca na base por EAN)
— Combobox Selecionar: Todas, Aumento, Reducao, PMC Nota, PMC ABC

Aplicacao:
— Opcao "Agora": atualiza ValorVenda/Markup/Projecao imediatamente em ProdutosDados
— Opcao "Na Finalizacao": salva sugestoes em ComprasProdutos (SugestaoVenda, SugestaoMarkup, SugestaoProjecao)
— Sugestoes aplicadas automaticamente ao finalizar a nota
Etapa 3: Conferencia
Campo bipe:
— Suporta multiplicador: 5*7896509970349 (5 unidades do barras)
— Busca no banco: barras principal + ProdutosBarras adicionais
— QtdeTotal = quantidade x fracao (persistida no banco)
— Fracao minima = 1 (correcao para registros antigos com fracao=0)

Sinalizadores por linha:
— ✓ Verde: conferido completo (QtdeConferida = Quantidade x Fracao)
— ⚠ Amarelo: conferido parcial (QtdeConferida > 0 mas menor que total)
— ✗ Vermelho: nao conferido (QtdeConferida = 0)

— Botao limpar conferencia (zera QtdeConferida de todos os itens)
Etapa 4: Finalizacao
Sub-etapa 1 — Duplicatas:
— Grid editavel com duplicatas da nota
— Checkboxes: Duplicatas Entregues, Nota Paga

Sub-etapa 2 — Lotes/Vencimentos:
— Grid editavel por produto: Lote e DataValidade
— Botao "Finalizar" executa toda a logica

Ao finalizar o sistema:
— Atualiza EstoqueAtual (soma Quantidade x Fracao)
— Atualiza UltimaCompra* (unitario, ST, outros, IPI, FPC do XML)
— Recalcula CustoMedio: ((estoque x CM) + (qtde x custo)) / (estoque + qtde)
— Aplica sugestoes de preco pendentes (SugestaoVenda/Markup/Projecao)
— Grava Lote e DataValidade em ProdutosDados
— Atualiza UltimaCompraEm e DataEntrada
— Registra log de auditoria por produto
— Status muda para "Finalizada"
Exclusao com reversao
— Nota pendente (Pre-Entrada/Precificacao/Conferencia): exclui cascade simples
— Nota finalizada: reverte estoque (subtrai Quantidade x Fracao), registra log de reversao
— DELETE /api/compras/id

Atualizacao de Precos
@if (isAberta('precos')) {
Visao Geral
Modulo para atualizacao de precos de venda dos produtos. Acesso pelo menu: Manutencao > Atual. Precos (/erp/atualizacao-precos). Suporta atualizacao via ABCFarma (arquivo JSON local ou futuramente via API) e atualizacao manual (pendente).
Conceitos fundamentais
Formacao de Preco: Define como o preco de venda e calculado
— Markup: valorVenda = custoBase x (1 + markup/100)
— Projecao de Lucro: valorVenda = custoBase / (1 - projecao/100). Maximo 99%.

Base de Calculo: Define qual custo e usado como referencia
— Custo da Compra: Soma de unitario + ST + outros + IPI + FPC + boleto + difal + frete
— Custo Medio: Media ponderada = ((estoqueAtual x custoMedioAtual) + (qtdeCompra x custoUnitario)) / (estoqueAtual + qtdeCompra)

Para produtos com PMC (ABCFarma): Valor de Venda = PMC. Markup e projecao sao recalculados automaticamente.

INCIDENTE - 10/04/2026

Ao remover migrations já aplicadas no banco (dotnet ef migrations remove) e criar uma nova migration consolidada, o EF Core gerou uma migration que DROPOU E RECRIOU as tabelas Vendas e VendaItens, causando perda total dos dados de vendas no banco.

Causa raiz: ao remover migrations, o snapshot do EF Core reverte para o estado anterior. Quando a nova migration é criada, o EF compara o modelo atual com o snapshot revertido e gera DROP + CREATE para as tabelas que "não existiam" no snapshot antigo.

Regras Obrigatórias

RegraMotivo
NUNCA remover migrations já aplicadas no banco Remove o snapshot, causando DROP + CREATE na próxima migration. Perda de dados.
Sempre criar migrations incrementais Cada mudança no modelo = uma nova migration. Nunca consolidar migrations existentes.
Revisar o conteúdo da migration ANTES de aplicar Abrir o arquivo .cs gerado e verificar se contém DROP TABLE. Se tiver, NÃO aplicar.
Só usar migrations remove para migrations NÃO aplicadas Se a migration ainda não foi aplicada no banco (não rodou MigrateAsync), pode remover com segurança.
Em produção: backup ANTES de qualquer migration Mesmo seguindo as regras, sempre fazer backup antes de aplicar migrations em produção.

Muitas migrations acumuladas Deixar. Não consolidar. O EF lida bem com centenas de migrations.

  3. Se não encontrou, e há um colaborador vendendo → busca hierarquia vinculada ao colaborador (HierarquiaDescontoColaborador)
  4. Se nenhuma específica → busca a hierarquia padrão (Padrao = true)
  5. Se nenhuma existe → venda sem desconto automático

  componente = item.Componente

  switch (componente):
    case Promocao (1):

        - ProdutoId = produto da venda
        - Ativo = true
        - DataHoraInicio <= agora <= DataHoraFim (ou sem fim)

Engine de Descontos (Integração com Hierarquia)
Ao adicionar um produto, o sistema consulta a engine de descontos para determinar o desconto aplicável.
Fluxo:
  1. Identificar hierarquia: Cliente → Convênio → Colaborador → Padrão
  2. Percorrer componentes: Na ordem configurada na hierarquia
  3. Primeiro match: O primeiro componente que retornar desconto é aplicado. Os demais são ignorados.
  4. Aplicação automática: Se a hierarquia tem AplicarAutomatico = true, o desconto é aplicado imediatamente no item.
  5. Desconto manual: Se não automático, o DESC.MAX é mostrado no rodapé para o operador aplicar manualmente.

Visão geral

O módulo fiscal emite NFC-e (Nota Fiscal de Consumidor Eletrônica) modelo 65 layout 4.00 diretamente para a SEFAZ. A implementação é manual (XML montado via StringBuilder), sem bibliotecas externas. O XML é assinado digitalmente com certificado A1 (PFX) e transmitido via SOAP.

Fluxo de emissão
1. Venda finalizada → POST /api/vendas/{id}/finalizar
2. Emissão NFC-e → POST /api/nfce/emitir/{vendaId}
3. NfceService monta XML, assina (RSA-SHA1), envia SOAP à SEFAZ
4. Resposta processada → protNFe/infProt/cStat (100=Autorizada)
5. NFC-e salva no banco (XmlEnvio + XmlRetorno + protocolo)
6. DANFE exibido em modal com botões Imprimir / WhatsApp / OK
Configurações necessárias (Configurações > Fiscal)
fiscal.regime.tributario — 1=Simples Nacional, 2=SN Excesso, 3=Regime Normal

fiscal.nfce.csc — Código de Segurança do Contribuinte (obtido na SEFAZ)
fiscal.nfce.csc.id — ID numérico do CSC
fiscal.nfce.serie — Série da NFC-e (default 1)
fiscal.nfce.numero.atual — Último número emitido (auto-incrementa)
Pré-requisitos
Certificado digital A1 (.pfx) cadastrado em Configurações > Certificado Digital
Código IBGE do Município preenchido na Filial
CSC e ID do CSC obtidos no portal da SEFAZ do estado
Produtos com NCM, CFOP e CSOSN/CST cadastrados (aba Fiscal do produto)
Estrutura do XML NFC-e
ide — Identificação (UF, série, número, data, ambiente, modelo 65)

Regras críticas aprendidas
Decimais: Sempre usar CultureInfo.InvariantCulture (ponto, não vírgula)
CSOSN: TrimStart('0') — SEFAZ espera "102" não "0102"
PIS/COFINS: Tag depende do CST (PISAliq para 01/02, PISNT para 04-09, PISOutr para demais)
vTotTrib: Primeiro elemento dentro de <imposto>, antes de <ICMS>
vPag: Soma(vPag) deve ser = vNF + vTroco (regra SEFAZ)
vTroco: Sempre presente, mesmo quando 0.00
infNFeSupl: Inserir antes da tag <Signature xmlns=...>, não dentro
QR Code PR: v3 formato http://www.fazenda.pr.gov.br/nfce/qrcode?p=CHAVE|3|AMBIENTE
SOAP: Envelope sem whitespace/indentação
NCM: Campo é CodigoNcm (não Codigo) na entidade Ncm
Retorno: Ler protNFe/infProt/cStat (status da nota), não cStat do lote

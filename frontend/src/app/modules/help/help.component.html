Sincronismo e Multifiliais (Arquitetura Completa)
@if (isAberta('sync-doc')) {
Visao Geral
Cada loja roda em servidor local (localhost + PostgreSQL local). O Railway e o banco central na nuvem com dados consolidados de todas as filiais. Fluxo: Filial faz CRUD local → PUSH para Railway → Railway aplica no proprio banco E enfileira na SyncFila → Outras filiais fazem PULL.
Fluxo do Sync (ciclo a cada 30s)
1. Autentica no Railway como usuario SISTEMA (senha rotativa diaria via SHA256). 2. PUSH: envia operacoes pendentes da SyncFila local para POST /api/sync/enviar. 3. Railway recebe, ORDENA por dependencia, APLICA no proprio banco (AplicandoSync=true), e enfileira para outras filiais. 4. PULL: filial busca GET /api/sync/receber?filialId=X&ultimoId=Y, ordena por dependencia, e aplica localmente.
Railway como Banco Consolidado
O Railway NAO e apenas um relay de mensagens. Ele APLICA cada operacao recebida no proprio banco PostgreSQL. Isso garante que o Railway tem TODOS os dados de TODAS as filiais. O banco do Railway serve para: consultas cross-filial, dashboards, apps externos. A flag AplicandoSync=true impede que a aplicacao gere novos registros na SyncFila (evita loop infinito).
Ordem de Processamento (Blindagem de Dependencias)
Operacoes sao ordenadas por nivel de dependencia ANTES de aplicar, evitando erros de FK. Para INSERT/UPDATE: pais primeiro (nivel menor). Para DELETE: filhos primeiro (nivel maior). A ordenacao e feita tanto no Railway (ao receber PUSH) quanto nas filiais (ao fazer PULL).
Niveis de Dependencia (SyncApplicator.GetOrdemTabela)
Nivel 0 (sem dependencia): Filiais, UsuariosGrupos, Ncms. Nivel 1 (depende de 0): Pessoas, Fabricantes, Substancias, GruposPrincipais, GruposProdutos, SubGrupos, Secoes, ProdutoFamilias, ProdutosLocais, NcmFederais, NcmIcmsUfs, NcmStUfs, UsuariosGruposPermissao. Nivel 2 (depende de 1): Colaboradores, Fornecedores, PessoasContato, PessoasEndereco, Produtos. Nivel 3 (depende de 2): Usuarios, ProdutosBarras, ProdutosMs, ProdutosSubstancias, ProdutosFornecedores, ProdutosFiscal, ProdutosDados. Nivel 4 (depende de 3): UsuarioFilialGrupos. Nivel 5 (locais): LogsAcao, LogsErro. Fallback desconhecido: 10.
Identificadores: IDs Globais por Faixa de Filial
Cada filial tem faixa exclusiva de 1 bilhao de IDs. Filial 1: 1.000.000.001+, Filial 2: 2.000.000.001+, Filial 3: 3.000.000.001+. Configurado automaticamente no startup via ALTER SEQUENCE RESTART WITH. FKs funcionam direto entre filiais (sem remapeamento). Colisao: praticamente impossivel.
SyncGuid (coluna auxiliar)
Toda entidade BaseEntity tem um campo SyncGuid (UUID, auto-gerado). NAO e PK, NAO tem FK. Serve para reconciliacao futura (ex: restaurar backup, resolver conflitos). Default no banco: gen_random_uuid(). Index criado em todas as tabelas.
Codigo Visivel
Formato "FilialCodigo.Sequencial" (ex: "1.523", "2.1089"). Gerado automaticamente por tabela via SequenciasLocais. Unico global. NAO gerado quando AplicandoSync=true (registros recebidos por sync ja vem com Codigo).
Regra Fundamental: Todas as filiais veem tudo
TODAS as filiais veem dados de TODAS as filiais. TUDO replica, incluindo Configuracoes, LogsAcao e LogsErro. Unicas excecoes: SyncFila e SequenciasLocais (infraestrutura de sync). DicionarioTabelas/Revisoes/Relacionamentos nao herdam BaseEntity, sao identicos pelo seed.
Tabelas Globais vs Por Filial
Globais (replicam para todas): Produto, ProdutoBarras, ProdutoMs, ProdutoSubstancia, Fabricante, GrupoPrincipal, GrupoProduto, SubGrupo, Secao, NCM, Substancia, Fornecedor, Pessoa, Colaborador, GrupoUsuario, Filial, Configuracoes, LogsAcao, LogsErro. Por Filial (replicam, dados variam por filial): ProdutoDados, ProdutoFiscal, ProdutoFornecedor, ProdutosLocais. NAO replicam (infraestrutura de sync): SyncFila, SequenciasLocais. Nao participam do sync (nao herdam BaseEntity): DicionarioTabelas, DicionarioRevisoes, DicionarioRelacionamentos.
ProdutoDados Multi-Filial
Ao criar um produto, o sistema cria automaticamente um registro ProdutoDados (zerado) para CADA filial ativa. Isso garante que toda filial tenha precos/estoque para editar. O SyncDados nunca deleta ProdutoDados de outra filial ao editar — apenas atualiza os que vieram no DTO.
Regra de Propagacao de Preco
Configuracao: produto.preco.regra (editavel em Configuracoes). Valores: "perguntar" = modal pergunta ao usuario quais filiais, "todas" = aplica automaticamente para todas, "atual" = apenas filial logada. Campos propagados: ValorVenda, ValorPromocao, Markup, ProjecaoLucro, DescontoMinimo, DescontoMaxSemSenha, DescontoMaxComSenha, Comissao, ValorIncentivo.
Tratamento de Erros no Sync
FK violation (23503): mensagem legivel indicando registro pai ausente. PK/Unique violation (23505): registro duplicado. Erros sao registrados na SyncFila local com campo Erro preenchido (visiveis no painel). O ponteiro AVANCA mesmo em falha para nao ficar em loop. Entries com erro sao detached do ChangeTracker.
Queda de Internet
Sistema funciona 100% offline. Operacoes ficam na SyncFila como pendentes (Enviado=false). Quando internet volta, envia tudo automaticamente no proximo ciclo. O Railway tambem acumula operacoes para filiais offline — ao reconectar, a filial puxa tudo pendente.
Arquivos-Chave do Sync
SyncApplicator.cs: logica compartilhada (AplicarOperacaoAsync, GetOrdemTabela, ResolverTipo, LimparNavigations, BuscarPorId). Usado pelo Railway e pelas filiais. SyncBackgroundService.cs: background service das filiais (Enviar/Receber a cada 30s). SyncController.cs: endpoints do Railway (POST enviar, GET receber, GET status, POST resetar, POST limpar). AppDbContext.SaveChangesAsync: intercepta CRUDs e gera SyncFila automaticamente (exceto quando AplicandoSync=true).
Checklist: Adicionar Tabela ao Sync
1. Adicionar no mapa _tiposPorTabela do SyncApplicator.cs (["NomeTabela"] = typeof(Entidade)). 2. Adicionar no GetOrdemTabela com o nivel correto de dependencia. 3. Se a tabela NAO deve replicar, adicionar na lista _tabelasSemSync do AppDbContext.cs. 4. Registrar no DatabaseSeeder o DicionarioTabela com Escopo e Replica corretos.
Tela Sincronismo
Mostra fila com filtros (periodo, status, tabela), paginacao, cards de status (pendentes, enviados, recebidos, erros). Permite forcar envio, resetar ponteiro de recebimento (rebuscar tudo), e limpar registros antigos.
}

Dicionario de Dados
@if (isAberta('dd')) {
Proposito
Documentacao viva da estrutura do banco. NAO altera o banco, apenas registra como deveria ser. Serve como base para implementar sync, validacoes e multifiliais.
Por tabela
Escopo (Global/Por Filial), Replica (Sim/Nao), Instrucao IA. Define o comportamento da tabela no sincronismo.
Por campo
Revisado, Obrigatorio, Unico, Instrucao IA, Observacao. Quando todos os campos estao revisados, a tabela fica com borda verde.
Backup JSON
Cada alteracao grava automaticamente em ContextDocuments/dicionario-dados.json. Vai pro Git como backup. Se perder o banco, importa pelo endpoint POST /api/dicionario-dados/importar.
}

Lançamento de Compras
@if (isAberta('compras')) {
Visão Geral
Módulo para lançamento de notas fiscais de entrada (compras de fornecedores). Permite registrar produtos comprados, atualizar estoque e preços automaticamente.
Entidades envolvidas
Compra — Cabeçalho da nota (fornecedor, número NF, data entrada, data emissão, valor total, status)
CompraItem — Itens da nota (produto, quantidade, valor unitário, desconto, valor total item, lote, validade)
CompraFinanceiro — Parcelas/duplicatas (vencimento, valor, forma pagamento, status)
Fluxo principal
1. Selecionar fornecedor (lookup)
2. Informar dados da nota (número, série, data emissão, data entrada)
3. Adicionar itens (busca produto por nome/barras, qtde, valores)
4. Sistema calcula totais automaticamente
5. Informar parcelas/financeiro
6. Ao confirmar: atualiza estoque, preços de última compra e custo médio
7. Gera log de auditoria completo
Regras de negócio
— Compra é sempre por filial (FilialId obrigatório)
— Ao confirmar compra, atualizar em ProdutoDados da filial:
  • estoqueAtual += quantidade comprada
  • ultimaCompraUnitario, ultimaCompraSt, etc.
  • custoMédio recalculado (média ponderada)
  • ultimaCompraEm = data da compra
— Compra pode ser: Pendente, Confirmada, Cancelada
— Compra confirmada não pode ser editada (apenas cancelada)
— Compra cancelada reverte estoque
Tabelas no banco
Compras — Id, FilialId, FornecedorId, NumeroNf, Serie, DataEmissao, DataEntrada, ValorTotal, ValorDesconto, ValorFrete, ValorOutros, Status, Observacao
CompraItens — Id, CompraId, ProdutoId, Quantidade, ValorUnitario, ValorDesconto, ValorSt, ValorIpi, ValorTotal, Lote, Validade
CompraFinanceiro — Id, CompraId, Parcela, Vencimento, Valor, FormaPagamento, Status
Checklist de implementação
1. Criar entidades Domain: Compra, CompraItem, CompraFinanceiro
2. Configurar DbContext (DbSets, Fluent API, índices)
3. Criar DTOs (List, Detalhe, Form)
4. Criar Service (CRUD + confirmar + cancelar)
5. Criar Controller com endpoints
6. Adicionar ao SyncApplicator (tabela + ordem de dependência)
7. Criar migration
8. Criar componente Angular (tela completa)
9. Adicionar rota e registro no menu/busca
10. Testar fluxo completo multi-filial
